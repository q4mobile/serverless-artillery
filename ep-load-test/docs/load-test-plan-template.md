## 1. Objective

Verify that the PSTN dial-out system can successfully connect a large number of participants (up to ~900) into a live conference meeting simultaneously, without degradation in call quality, IVR responsiveness, or system stability.

Specifically, we want to confirm that:
- All participants receive and answer the outbound call
- The IVR correctly prompts for and accepts meeting ID and PIN via DTMF
- All participants reach `CONNECTED` state within acceptable time
- The hand-raise toggle flow works under load
- No errors are produced by the Chime SMA, Lambda handlers, or DynamoDB

---

## 2. Scope

**In scope**

- Outbound PSTN dial-out via Chime SMA (`CreateSipMediaApplicationCall` / `UpdateSipMediaApplicationCall`)
- IVR DTMF flow (meeting ID + PIN entry) and DynamoDB participant state polling
- Hand-raise toggle (`*1#`) where exercised in the Artillery scenario
- Post-test participant summary from `npm run report` (run-state / peak state)

**Out of scope**

- **Inbound PSTN** (`NEW_INBOUND_CALL`, analyst dials in) — this runbook covers **dial-out** only; see **Tech plan crosswalk** for program-level inbound load.

## 3. Test Environment

| Component | Production value |
|---|---|
| AWS Region | `us-east-1` |
| Chime SMA (test) | `LOAD_TEST_SMA_ID` (dedicated test SMA — never the production SMA) |
| Caller ID | `LOAD_TEST_FROM_PHONE` (E.164) |
| Called number | `LOAD_TEST_TO_PHONE` (E.164, PSTN number the SMA answers on) |
| DynamoDB table | `events-streaming-serverless-conference-participants-prod` |
| Load generator | Artillery running locally on a Mac / CI machine |
| Production SMA guard | Set `PRODUCTION_SMA_ID` — processor will refuse to start if the test SMA matches it |

> **Important:** The test SMA is a dedicated AWS Chime SIP Media Application that mirrors production behaviour but is entirely separate from the production SMA. It must be deployed and pointing to the correct Lambda before the test.

## 4. Load Profile

| Parameter | Small run (smoke) | Medium run | Full run |
|---|---|---|---|
| Meetings (parallel) | 1 | 1–2 | **4** |
| Participants | 20 | 100 | **900** (225 per meeting) |
| Artillery load curve | Prefer `arrivalCount` + `duration` (exact N starts, spread over ramp window) | Same | Same |
| Full-run example | — | — | `duration: 300`, `arrivalCount: 900` (5 min ramp of starts; matches default YAML) |
| Estimated total test time | ~10 min | ~25 min | ~90 min (depends on scenario `think` / hangup duration) |

Total **scenario starts** across all Artillery `phases` must equal the number of **data** rows in `data/analysts-payload.csv` (see `tests/dial-out-payload-example.yml` and README — prefer `arrivalCount` + `duration` for an exact match). Otherwise the same attendee is dialled twice, which corrupts results.

## 7. Execution Steps

### 7.1 — Pre-registration (Day before the test, or same morning)

Register all participants for the target meeting(s) and generate the payload CSV.

```bash
cd ep-load-test

# For a single meeting

ANALYST_REGISTRATION_PASSWORD="<registration_password>" \
EP_API_BASE_URL="https://attendees.events.q4inc.com/rest/v1" \
npm run register-analysts
```

Or using a plan file for **4 parallel meetings × 225 analysts each** (`data/registration-plan.json`; copy from `data/registration-plan.example.json` and replace placeholder `meetingId` values):

```json

  { "meetingId": 111111111, "analystCount": 225, "registrationPassword": "<secret>" },
  { "meetingId": 222222222, "analystCount": 225, "registrationPassword": "<secret>" },
  { "meetingId": 333333333, "analystCount": 225, "registrationPassword": "<secret>" },
```

Verify output: `data/analysts-payload.csv` exists and data row count matches total scenario starts configured in Artillery `phases`.

### 7.2 — Pre-test verification (30 minutes before)

```bash
# Confirm Lambda resource policy is present
aws lambda get-policy \
  --function-name events-streaming-prod-conferenceEventHandler \
  --region us-east-1 \
  | jq '.Policy | fromjson | .Statement[] | select(.Principal.Service == "voiceconnector.chime.amazonaws.com")'
```

If the policy is missing, add it:

```bash
aws lambda add-permission \
  --function-name events-streaming-prod-conferenceEventHandler \
  --statement-id AllowChimeSMAInvoke \
  --action lambda:InvokeFunction \
  --principal voiceconnector.chime.amazonaws.com \
  --region us-east-1
```

Update `tests/dial-out-payload-example.yml` to match the participant count, for example:

```yaml
phases:
  - duration: 300
    arrivalCount: 900
    name: "900 dial-outs (4×225) evenly over 5 minutes"
```

---

### 7.3 — Execute the test

```bash
cd ep-load-test

LOAD_TEST_SMA_ID="<test-sma-id>" \
LOAD_TEST_FROM_PHONE="+1XXXXXXXXXX" \
LOAD_TEST_TO_PHONE="+1XXXXXXXXXX" \
DIALOUT_PARTICIPANTS_TABLE_NAME="events-streaming-serverless-conference-participants-prod" \
PRODUCTION_SMA_ID="<prod-sma-id>" \
npx artillery run tests/dial-out-payload-example.yml
```

Monitor Artillery output in the terminal. Artillery prints a rolling summary every 10 seconds including errors, scenarios launched, and scenarios completed.

Monitor CloudWatch simultaneously:
- Lambda error rate (`conferenceEventHandler`, test SMA Lambda)
- DynamoDB consumed capacity and throttles
- Amazon Polly invocation throttles

## 9. Rollback / Abort Plan

If the test must be stopped early:

1. Press `Ctrl+C` in the Artillery terminal — Artillery will stop launching new virtual users. In-flight calls will complete their current step and then call `hangUpOnError` to terminate open Chime calls.
2. If calls remain open after ~2 minutes, manually terminate them:
   ```bash
   aws chime-sdk-voice delete-sip-media-application-call \
     --sip-media-application-id <test-sma-id> \
     --transaction-id <txn-id> \
     --region us-east-1
   ```
3. The test SMA has no impact on the production SMA — production calls are unaffected.
4. DynamoDB records written during the aborted test are harmless; they do not affect future runs.

## 11. Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Amazon Polly throttling causes DTMF prompts to fail | Medium | High | Run during off-peak; monitor CloudWatch `ThrottledRequests`; lengthen ramp or lower peak `arrivalRate` / `rampTo` if needed |
| DynamoDB GSI read throttles under high participant load | Medium | Medium | Ensure on-demand capacity mode; monitor metrics |
| Chime SMA Lambda cold starts delay DTMF responses | Low | Medium | Pre-warm Lambda with a small run before the full test |
| Wrong SMA ID used (test vs production) | Low | Critical | `PRODUCTION_SMA_ID` guard in processor; double-check IDs in env |
| CSV row count mismatch with Artillery phase totals | Low | Medium | Script prints row count; verify `arrivalCount` or ramp formula before starting |
| Participant CSV contains stale/expired registrations | Medium | High | Always regenerate CSV on the day of the test |
